1. Overview

1. 概述

AgentProfile is a JSON-based open standard for defining AI agents. Each profile is a self-contained document that fully describes an agent's identity, behavior, capabilities, and resources. The core principle is simple:

AgentProfile 是一个基于 JSON 的开放标准，用于定义 AI Agent。每个 Profile 是一个自包含的文档，完整描述了 Agent 的身份、行为、能力和资源。核心理念很简单：

One sentence in, one complete agent out.

一句话输入，一个完整 Agent 输出。

A user provides a single natural language instruction. The Agent Constructor analyzes it and produces a structured AgentProfile document containing everything needed to instantiate a working agent.

用户提供一句自然语言指令，Agent 构造器分析指令并生成一份结构化的 AgentProfile 文档，其中包含创建可用 Agent 所需的一切。

3. Field Definitions

3. 字段定义

An AgentProfile document contains seven fields. Three are required, four are optional.

一个 AgentProfile 文档包含七个字段。三个必填，四个可选。

name string Required

A concise name that captures the agent's core role. Should be immediately understandable and unique within a deployment context.

简洁的名称，体现 Agent 的核心角色。应当一目了然，在部署环境中具有唯一性。

  • Keep it short: typically 3-10 words
  • Use the agent's domain and function as key elements
  • Avoid generic names like "AI Assistant" or "Helper"
  • 保持简短：通常 3-15 个字
  • 以 Agent 的领域和功能作为核心要素
  • 避免使用泛化名称，如"AI 助手"或"帮手"
description string Required

A single sentence summarizing the agent's role and primary capabilities. This is what users see when browsing available agents.

一句话概括 Agent 的角色和主要能力。这是用户浏览可用 Agent 时看到的内容。

  • Limit to one sentence (under 150 characters recommended)
  • Start with the agent's function, then describe its approach
  • Be specific about what value the agent provides
  • 限制为一句话（建议不超过 80 个字）
  • 先说 Agent 的功能，再描述其方法
  • 明确说明 Agent 提供的价值
details string Required

A Markdown-formatted string containing the agent's complete behavioral specification. This is the most important field — it defines how the agent thinks and acts. Must follow the Details Structure.

Markdown 格式的字符串，包含 Agent 的完整行为规范。这是最重要的字段 — 它定义了 Agent 如何思考和行动。必须遵循 Details 结构。

  • Must contain four sections: Role, Core Dimensions, Standards, Output Format
  • Use standard Markdown syntax (headings, tables, lists)
  • Newlines in JSON are encoded as \n
  • 必须包含四个部分：角色定位、核心维度、标准规范、输出格式
  • 使用标准 Markdown 语法（标题、表格、列表）
  • JSON 中的换行符编码为 \n
agent_template string Optional

Specifies which runtime template to use when instantiating the agent. Defaults to "default". Templates control the agent's execution environment, prompt assembly strategy, and interaction patterns.

指定实例化 Agent 时使用的运行时模板，默认为 "default"。模板控制 Agent 的执行环境、提示词组装策略和交互模式。

tools string[] Optional

An array of tool identifiers sourced from the tool repository. Each identifier references a specific tool the agent can invoke during interaction.

从工具仓库中获取的工具标识符数组。每个标识符引用一个 Agent 在交互中可以调用的工具。

  • Each entry must match a valid tool identifier in the tool repository
  • Empty array [] means the agent uses no tools
  • Order does not matter
  • 每个条目必须匹配工具仓库中的有效工具标识符
  • 空数组 [] 表示 Agent 不使用工具
  • 顺序无关
skills string[] Optional

An array of skill module names sourced from the skill repository. Skills are pre-built knowledge/behavior modules that augment the agent's capabilities in specific domains.

从技能仓库中获取的技能模块名称数组。技能是预构建的知识/行为模块，在特定领域增强 Agent 的能力。

subagents string[] Optional

An array of sub-agent names that this agent can delegate tasks to. Enables multi-agent collaboration patterns where a primary agent orchestrates specialized sub-agents.

此 Agent 可以委派任务的子 Agent 名称数组。支持多 Agent 协作模式，由主 Agent 编排专业化的子 Agent。

4. Details Structure

4. Details 结构

The details field is the behavioral core of an AgentProfile. It must be a Markdown string containing exactly four sections, each as a level-2 heading (##).

details 字段是 AgentProfile 的行为核心。它必须是包含四个部分的 Markdown 字符串，每个部分使用二级标题（##）。

Section 1: Role

第一部分：角色定位

Defines who the agent is. Covers identity, domain expertise, communication style, and guiding principles.

定义 Agent 是谁。涵盖身份、领域专长、沟通风格和指导原则。

details — Role
## Role

As a [domain] [function] specialist, provide [approach style]
assistance to help users [primary goal].

Focus on [key principle]. Prioritize [quality attribute]
over [trade-off].

What to include: domain scope, persona tone (formal/casual/encouraging), core methodology, constraints and boundaries.

应包含：领域范围、人设语气（正式/随意/鼓励式）、核心方法论、约束与边界。

Section 2: Core Dimensions

第二部分：核心维度

Defines what the agent focuses on. A structured breakdown of the agent's attention areas. Use tables for multi-dimensional analysis or lists for sequential focus areas.

定义 Agent 关注什么。Agent 注意力领域的结构化分解。使用表格进行多维分析，或使用列表进行顺序关注点。

details — Core Dimensions
## Core Dimensions

| Dimension        | Focus Points                           |
|------------------|----------------------------------------|
| **Area 1**       | Specific concerns, checks, criteria    |
| **Area 2**       | Specific concerns, checks, criteria    |
| **Area 3**       | Specific concerns, checks, criteria    |

What to include: 3-8 dimensions, each with concrete focus points. Optionally add a "common pitfalls" or "warning" column.

应包含：3-8 个维度，每个维度有具体的关注要点。可选添加"常见陷阱"或"警告"列。

Section 3: Standards

第三部分：标准规范

Defines the quality bar. References to industry standards, evaluation criteria, severity levels, and feedback methodology.

定义质量标准。引用行业标准、评价标准、严重等级和反馈方法。

details — Standards
## Standards

- Follow [industry standard / framework / guideline]
- Classify issues by severity: critical / warning / suggestion
- Always explain the reasoning behind each recommendation
- Provide actionable, specific feedback over vague commentary

What to include: reference standards, severity taxonomy, feedback tone requirements, quality thresholds.

应包含：参考标准、严重性分类、反馈语气要求、质量阈值。

Section 4: Output Format

第四部分：输出格式

Defines how the agent structures its responses. A numbered list of response sections that the agent should follow consistently.

定义 Agent 如何组织其响应。Agent 应一致遵循的响应结构编号列表。

details — Output Format
## Output Format

1. **Summary**: High-level assessment and key findings
2. **Analysis**: Detailed breakdown by dimension
3. **Recommendations**: Prioritized, actionable suggestions
4. **Next Steps**: Follow-up actions or resources

What to include: 3-6 response sections, each with a bold title and purpose description. Keep it prescriptive so outputs are consistent.

应包含：3-6 个响应部分，每部分有加粗标题和用途说明。保持规范性以确保输出一致。

5. Tool & Skill Repositories

5. 工具与技能仓库

AgentProfile separates agent definitions from their capabilities. Tools and skills live in external repositories and are referenced by identifier.

AgentProfile 将 Agent 的定义与其能力分离。工具和技能存放在外部仓库中，通过标识符引用。

Tool Repository

工具仓库

The tool repository stores callable tools — actions the agent can invoke during interaction (e.g., web search, file operations, API calls, code execution). Each tool has a unique identifier and a well-defined interface.

工具仓库存储可调用的工具 — Agent 在交互中可以调用的动作（如网页搜索、文件操作、API 调用、代码执行）。每个工具有唯一标识符和明确定义的接口。

Skill Repository

技能仓库

The skill repository stores knowledge and behavior modules — pre-built packages of domain expertise, prompt fragments, or specialized instructions. Skills augment the agent's knowledge without requiring tool invocation.

技能仓库存储知识与行为模块 — 预构建的领域专业知识包、提示词片段或专业化指令。技能增强 Agent 的知识，无需工具调用。

Design Principle

Keep profiles declarative. The profile says what tools and skills the agent needs. The runtime decides how to load and inject them.

设计原则

保持 Profile 的声明式特性。Profile 声明 Agent 需要什么工具和技能，运行时决定如何加载和注入它们。

6. Agent Constructor

6. Agent 构造器

The Agent Constructor is the engine that transforms a one-sentence instruction into a complete AgentProfile. It performs three sequential steps:

Agent 构造器是将一句话指令转化为完整 AgentProfile 的引擎。它执行三个顺序步骤：

  1. Generate Profile — Analyze the instruction and produce name, description, and details fields with a complete four-section behavioral specification.
  2. Match Tools — Scan the tool repository, evaluate relevance to the described scenario, and populate the tools array with matching identifiers.
  3. Match Skills — Scan the skill repository, evaluate relevance, and populate the skills array with matching module names.
  1. 生成 Profile — 分析指令并生成 name、description 和 details 字段，包含完整的四段式行为规范。
  2. 匹配工具 — 扫描工具仓库，评估与描述场景的相关性，将匹配的标识符填入 tools 数组。
  3. 匹配技能 — 扫描技能仓库，评估相关性，将匹配的模块名填入 skills 数组。

The constructor also fills default values: agent_template is set to "default", and subagents is set to [].

构造器还会填充默认值：agent_template 设为 "default"，subagents 设为 []。

7. Complete Example

7. 完整示例

Input instruction: "A Python code review assistant"

输入指令："Python 代码审查助手"

python-code-review.json
{
  "name": "Python Code Review Assistant",
  "description": "An AI assistant focused on Python code review, providing style checks, bug detection, security scanning, and performance optimization suggestions.",
  "details": "## Role\n\nAs a Python code review specialist, provide thorough, constructive feedback with clear explanations. Focus on helping developers write cleaner, safer, and more performant code while following community best practices.\n\n## Core Dimensions\n\n| Dimension | Focus Points |\n|-----------|-------------|\n| **Style & Formatting** | PEP 8 compliance, naming conventions, code readability |\n| **Bug Detection** | Logic errors, off-by-one, null references, race conditions |\n| **Security** | Input validation, SQL injection, path traversal, dependency vulnerabilities |\n| **Performance** | Algorithm complexity, unnecessary allocations, I/O bottlenecks |\n| **Maintainability** | DRY principle, single responsibility, appropriate abstractions |\n\n## Standards\n\n- Follow PEP 8 and PEP 257 style guidelines\n- Reference OWASP Top 10 for security concerns\n- Provide severity levels: critical / warning / suggestion\n- Always explain the \\\"why\\\" behind each recommendation\n\n## Output Format\n\n1. **Summary**: Overall code quality assessment and key findings\n2. **Issues**: List of issues with severity, location, and explanation\n3. **Suggestions**: Concrete code improvements with before/after examples\n4. **Highlights**: Notable good practices found in the code",
  "agent_template": "default",
  "tools": [],
  "skills": ["python-lint", "security-scan", "complexity-analysis"],
  "subagents": []
}

8. Validation Rules

8. 校验规则

A valid AgentProfile document must satisfy all of the following:

有效的 AgentProfile 文档必须满足以下所有条件：

RuleRequirement
R1Root must be a JSON object
R2name must be a non-empty string
R3description must be a non-empty string
R4details must be a non-empty string containing ## headings for all four sections
R5If present, agent_template must be a non-empty string
R6If present, tools must be an array of strings
R7If present, skills must be an array of strings
R8If present, subagents must be an array of strings
R9No additional top-level keys beyond the seven defined fields
规则要求
R1根必须是 JSON 对象
R2name 必须是非空字符串
R3description 必须是非空字符串
R4details 必须是非空字符串，包含四个部分的 ## 标题
R5如存在，agent_template 必须是非空字符串
R6如存在，tools 必须是字符串数组
R7如存在，skills 必须是字符串数组
R8如存在，subagents 必须是字符串数组
R9不允许超出七个定义字段之外的顶层 key

9. Best Practices

9. 最佳实践

Do

Write details as if briefing a new team member. Be specific about what to focus on, what standards to follow, and how to structure output.

推荐

撰写 details 时应像在向新团队成员做简报。明确说明关注什么、遵循什么标准、如何组织输出。

Do

Use tables in Core Dimensions for multi-faceted analysis. Tables make the agent's focus areas scannable and unambiguous.

推荐

在核心维度中使用表格进行多维分析。表格使 Agent 的关注领域一目了然且无歧义。

Do

Keep skills and tools minimal. Only include what the agent genuinely needs. Fewer, well-chosen capabilities outperform a bloated toolset.

推荐

保持技能和工具精简。只包含 Agent 真正需要的。少而精的能力优于臃肿的工具集。

Avoid

Don't write vague details like "be helpful and accurate." Every statement in details should be specific enough to change the agent's behavior.

避免

不要写模糊的 details，如"保持有帮助且准确"。details 中的每一条都应具体到能够改变 Agent 的行为。

Avoid

Don't mix concerns across sections. Role defines identity, Dimensions define focus, Standards define quality, Output defines structure. Keep them separate.

避免

不要跨部分混淆关注点。角色定义身份，维度定义关注点，标准定义质量，输出定义结构。保持各部分独立。

10. Versioning

10. 版本管理

The AgentProfile specification follows Semantic Versioning. The current version is v1.0.

AgentProfile 规范遵循语义化版本号。当前版本为 v1.0。

  • Patch (v1.0.x): Clarifications and typo fixes. No schema changes.
  • Minor (v1.x): New optional fields or sections. Fully backward compatible.
  • Major (vX.0): Breaking changes to required fields or structure.
  • 补丁版本 (v1.0.x)：澄清和修正。无 schema 变更。
  • 次版本 (v1.x)：新增可选字段或部分。完全向后兼容。
  • 主版本 (vX.0)：必填字段或结构的破坏性变更。
